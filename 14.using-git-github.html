Using Git with GitHub

This guide will show you how to install Git and then use Git with GitHub. This is not a complete Git tutorial but is a very good start for someone who has not used it before. There is a very good online book for using Git. If you are new to Git, reading the first three chapters will be well worth your time. See the 'See Also' section at the bottom of the post for a link to the book.

There are different ways you can create a repository and work with it. The process below is one possible process and one that I think it is easy for a new Git/GitHub user to follow and learn.

Like many specialized subjects, Git has its own vocabulary and understanding Git well enough to make use of it requires that you learn its vocabulary. Here is what some of the words & phrases in the 'git status' results mean:

master
Git is like a tree. You can create branches to you tree. The default branch is called 'master'. Look at your GitHub home page and find the dropdown that says "Branch: master"
origin
'origin' is a name has associated with the repository on your local machine. As we will see later, it holds the URL to the repository on GitHub.
Your branch is up-to-date with 'origin/master'
This message means that the repository on your local machine matches the repository on GitHub. It does not mean that everthing in your project directory matches what is on GitHub. You probably need to understand 'commit' before you can understand this. So come back and read this again once you undertand 'commit'.
'nothing to commit'
In a simple sense this means, 'you have nothing ready to make permenant in the repository'. In Git language it says you have nothing in the repository's staging area.
'working directory is clean'
This means there is nothing in your project's directory that is not already in the Git repository on you local machine. It is not saying anything about the repository on GitHub.

Here are some definitions to help with the next step. These are 'plain-language' definitions, not official Git definitions.

Untracked
An untracked file has never been added to Git.
Committed
A file that has been put into Git using $ git commit
Staged
A staged file is ready to be added to Git. It is in the 'staging area' but has not been 'committed' yet. A file is staged with the $ git add command. You can use $ git add on a new file or a perviously committed file that has been since been changed.

$ git push Explained

Someone recently sent around an articale that said to people writing tutorials, 'Never say anything is simple'. I think this is great advice. After you have done something complex enough to fully understand it, it does seem simple. But if you are new to something as complex and non-intuitive as Git it isn't simple at all. So I am not going to say $ git push is simple. I will say that if you put aside the need to understand what it did, using $ git push to move changes to GitHub was pretty easy. It is only one command to remember and if you are feeling confused I think it is worth taking a moment to realize that we only used 3 commands, add, commit and push.

'origin' and 'remote'

To understand origin you need to understand 'remote'. Remote refers to the repository on GitHub. Execute the below command.

$ git remote -v

Given the results in the screenshot it appears that 'origin' is a name for something that holds a URL, and it is. It is the name of the Git repository on GitHub, but not the name you gave it. Git assigned this name when you did the clone. You can change it if you want to but for the moment there is no need to.

You'll note that there is 'fetch' and 'push'. These are the URLs that Git will use to fetch data from and push data to GitHub.

master -> master

This means we move date from the master branch in our local repository to the master branch on the remote repository (i.e., the repository on GitHub).

Someone told me recently that my use of the command line was old school and that people today use GUI tools. But I follow several developer groups on Google+ and and the attitued is often expressed that real developers use the command line.

My own take is that you should do what work for you but ..., if you are a software developer, you should know how to use the command line. In the 25+ years since I started making software I have never met a developer who could not use the command line.

For Git I do all my commits with the command line. I will look at small changes between uncommited and the last commit via command line as well. But for extensive changes or changes between to commits I use a GUI tool.

Along these same lines, here is an admission: Although I do many commits per day, I don't understand everything in the $ git push status message. It would be nice to, and eventually I'm sure I will, but understanding everything isn't necessary to effective use Git.
