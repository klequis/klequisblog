This tutorial will introduce you to the command line and along the way will teach you about the file permission system that is common to Linux and Mac. It is part of the series If you following the series and already know how to use the command line, feel free to skip this tutorial. If you are new to the command line read on.

Why Use the Command Line?

If you are a general computer user creating spreadsheets, editing files, managing music, surfing the web, etc., you can use Ubuntu GNOME or any of the other major OSs without the command line. As a developer, this is not the case. As I meet more aspiring developers the question of how well you need to know the command line or if you really need to use it comes-up often. Let's take a look at some developer documentation for clues.

  • Installing Node.js via package manager
  • Installing npm packages locally
  • Webpack Getting Started guide

I will also note that I have yet to meet an experienced developer, and I have been meeting them for a very long time, who could not use the command line, and I think from the perspective of managing how other's view your developer skills, you should be comfortable with the command line or you may be viewed as not very technical. What other's think of your software development skills matters a lot when you want to find a work or get a promotion.

Use a Cheatsheet

It will help to have a cheatsheet of commands. This can be a text file or it can go into a program such as Evernote or OneNote. Keep it simple so you can add to and refer to it quickly. A cheatsheet listing of all the commands used in this guide is at the bottom of the post.

There is a way to avoid using the command line most of the time. I am strongly biased against using it but I'm writing this series to inform, not indoctrinate, so here it is.

sudo nautilus

Using this command will open Files with root privileges. It is very powerful and dangerous as you can change pretty much everything, even things that will destroy your system. I don't use it and I advise you to learn to use the command line and gain the understanding of the Ubuntu OS that comes with it.

Wherever you see $ type what follows it into terminal.

For example, if you see $ ls -la, you type into terminal "ls -la" exactly as you see, paying careful attention to every character, especially spaces and capitalization.

  1. Everything you do in Terminal is case sensitive. If something doesn't work, be sure to double check the case.
  2. You can open multiple tabs in terminal by pressing Ctrl-t.
  3. Spaces in file names require quotes. It is easiest to not use spaces in your file names.
  4. You can see all of your previous commands in Terminal by using your up and down arrow keys. Doing so will save you a lot of typing and many typos.

Definitions

Terminal
An application that gives you command line access to the operating system.
Command prompt
A symbol displayed which indicates the computer is ready for input. For Ubuntu, the command prompt symbol is $.
Command line
Everything you enter to the right of the command prompt.
User
'User' can be a name representing a person who uses the computer or an abstract user such as 'cdrom', which is a group containing users who can use the computer's CD-ROM.
Group
A container that holds one or more users. Groups have a set of permissions and when a user is added to the group they get those permissions.
Superuser
A special account/user who is the ultimate administrator. The superuser can make any and all changes to the system including destroying it.

Getting Started with the Command Line

In Ubuntu GNOME you access the command line via an application named Terminal. To launch Terminal press Super, type 'terminal' and press Enter.

Terminal will be launched with the current directory as your home directory. Let's dissect the command prompt.

karl @ karl-g3 : ~ $

karl User name
@ Just "at"
karl-g3 Machine/computer name
: Just a "colon"
~ "tilda" is a short-hand for your "home" directory. My home directory is /home/karl.
$ Indicates commands will be executed with your privileges and not the superuser's privileges (which would be indicated with '#'). This is explained in more detail below.

'karl@karl-g3:~$' means the user is 'karl', the machine name is 'karl-g3', the current directory is '~' (which is short for /home/karl) and the system is ready for input '$'.

Using the command to change directory $ cd Downloads changes the current directory to /home/karl/Downloads. Note that the 'Downloads' directory name is capitalized and it must be so on the command line as well.

Setup your windows

If you have the screen real estate then open Terminal & Files side-by-side so that as you type commands into Terminal you can see the changes in Files.

The file manager in GNOME is simply called "Files".

The First Command: $ ls

$ ls

This command will show the contents of the current directory.

You can add options to the ls command to change the output. The most common are -l and -a.

"-l" (lower case L) shows a more detailed list view of the directory contents. We will analyze the parts of this detail shortly.

$ ls -l

"-a" shows hidden files and directories. The filename of hidden files & directories starts with a period.

$ ls -la

Permissions

In explaining the output of the ls command I will go through a long explanation of Linux permissions. If you have never been exposed to this before it is likely you will not fully understand it. Worry not, it takes some time and review of different explanations to fully comprehend. Give the section a read and then move on. Conquer Linux permissions another day.

ls -l, ls -la Output explained

There is a great deal of detail printed out by the ls command. We will go through it piece by piece using '-rw-rw-r-- 1 karl karl 0 May 28:08:50 test1.txt' as an example. Each of the first 10 letters have individual meaning. The first character indicates the file type and the next 9 show the permissions. The meaning of the the individual characters is listed below. Although it is potentially confusing having a user named karl and a group named karl, Ubuntu creates a group for each user with the same name as the users login name.

A user is assigned privileges which are the right to perform an action on a computer. 'Login' is the right to log in. 'Read' is the right to read (aka view/see) a file, directory or other object on the computer.

A group is a container of users. Members of the group are given privileges. Ubuntu comes with a set of default groups. Groups can be created, deleted and renamed. Users and privileges can be added to and removed from a group.

Character Meaning
- (hyphen) When in the first position '-' indicates that the item is a file.
When in any of the following 9 positions it means 'the permission is not granted'.
d When in the first position it indicates that the item is a directory (aka folder).
l (lower case L) When in the first position indicates the item is a symbolic link.
r Read permission. The user can see and open the file.
w Write permission. The user can edit the file.
x Execute permission. The user can execute the file as a program or other executable. In Linux, all folders must have execute.

Output Parts

- rw- rw- r-- 1 karl karl 0 May 28 08:50 test1.txt

Filetype Owner's permissions Group's permissions Other's permissions Link count
Owner Group File size Last access date/time Filename

Characters 2 through 10 are groups of 3. Each group of 3 applies to a specific type of system user. Possible values for the positions within a group of 3 are:

Position 1 of 3"r" for read or "-" for 'does not have read permission'
Position 2 of 3"w" for write or "-" for 'does not have write permission'
Position 3 of 3"x" for execute or "-" for 'does not have execute permission'
RefPartMeaning
A-"-" (hyphen) indicates it is a file and not a directory. This position could also be "d" for directory. There are other designators that can exist in this position. See the references at bottom of this post for more detail.
Brw-The first grouping of three (characters 2-4) is the permissions for the user. In this case user karl has read & write but not execute.
Crw-The second grouping of three (characters 5-7) is the permissions for the group owner. In this case the group karl has read & write but not execute.
Dr--The third grouping is the permissions for everyone else. In this case everyone other than user karl and members of the group karl, have read but not write and not execute.
E1The number of links to this file. An explanation of this is out of scope for this post.
FkarlThe user who is the "Owner" of the item. It will be an individual user account. In this case I am logged-in as karl and kar is the owner.
Gkarl"Group owner". There are many groups on your Ubuntu GNOME machine and one of them is named with your user name.
H0This size of the item. This is an empty text file and the number is rounded, so it is showing as zero.
IMay 28 08:50The date & time.
Jtest1.txtThe name.

Creating Files $ gedit,$ >,$ touch

Up to now we have been working with directories. Now let's work with some files while exploring several different methods to create files.

Overall I love GNOME. However, one of my frustrations with it is that you can not right-click in Files to create a new document. I'm sure there is a way to fix that and finding it is on my to do list.

Most of the time I create files from within a text editor. I currently use Atom. Knowing how to create files from the command line is useful but you can decide what workflow works best for you.

gedit - is a common text editor that comes with Ubuntu.

Make sure you are in '~/workspace/samplefiles'. If not then execute the below command. (remember '~' means 'home').

$ cd ~/workspace/samplefiles

The following command will launch gedit with a new file open. 

$ gedit page1.html

Type the below into page1.html.

<h1>page1.html</h1>

Save (Ctrl+s) the file and quit (Ctrl+q) gedit. Then check-out page1.html in Files.

Using Terminal and the $ ls -l command, look in '~/workspace/samplefiles' for page1.html.

'>' Redirection - You can read-up on redirection on your own. For now I'll just note that normally output will go to the screen but you can 'redirect' it using '>'. First lets try to print something without redirection.

$ echo '<h1>page2.html</h1>'

That printed to the screen. Now lets redirect it to a file.

$ echo '<h1>page2.html</h1>' > page2.html

View the file in Files and then in Terminal.

Open the file with FireFox.

$ firefox page2.html

You can also create empty files with redirection.

$ > page3.html

$ touch - touch is an easy way to create new empty files with the command line. It is also used to change the date/time of a file.

$ touch file1.css

$ rm - Removing files and folders

There is a rmdir command but I don't find it necessary. You can remove (aka delete) any file or directory using the rm command along with a the option -r for 'recursive'. We will also discuss -f for force.

-r (recursive) will drill down the directory structure and remove everything below the specified node.

-f (force) ignores nonexistent files, missing operands and never prompts the user. What that really means is it will silently go about removing things and not present any warnings or other information to you. I'm tempted to say you should be sure you know what you are deleting before using -f but the truth is, if you are unsure you should not delete anything until you are, and once you are -f will supress messages you don't need to see.

Let's setup sample files for these exercises. First, clean-up the existing /samplefiles directory.

$ cd ~/workspace
$ rm -rf samplefiles
$ mkdir samplefiles
$ touch file1.txt
$ mkdir emptydir
$ mkdir non-emptydir
$ cd non-emptydir
$ touch file2.txt
$ touch file3.txt
$ mkdir innerdir
$ cd ..

Take a moment to explore your new files and directories.

Issue the ls -l command after each step so you can see what changes.

Remove file1.txt.

$ rm file1.txt
$ ls -l

$ rm without -r cannot delete a directory. Try it.

$ rm emptydir

Adding -r will make it work.

$ rm -r emptydir
$ rm -r non-emptydir
$ ls -l

Now let's see how -f works. First create 2 files that will be read-only to you. ('sudo' will be explained in the next section.)

$ sudo touch file4-root.txt
$ sudo touch file5-root.txt
$ ls -l

Note that the files are owned by 'root root'. In this case, your permissions are the last group of 3 (everyone else) which are 'r--' meaning read, no write and no execute.

If you use $ rm without -r you will be asked to confirm you want to delete the file. If you use -r you will not be asked.

$ rm file4-root.txt
$ rm -f file5-root.txt

$ sudo

sudo (superuser do) is a utility for UNIX- and Linux-based systems that provides an efficient way to give specific users permission to use specific system commands at the root (most powerful) level of the system. Sudo also logs all commands and arguments. Source: TechTarget

So far we have been working with files owned by the currently log-in user (you). If you want to work with files and directories that you don't own you need to use sudo. Let's create both a directory and a file with sudo and then view the results.

$ cd ~/workspace/samplefiles
$ sudo mkdir sudorootdir

You will be prompted for your password, the same one you log in with.

You can't use $ sudo > ... so we will use touch to create the file.

$ sudo touch sudorootfile.txt

Performing a directory listing will show that both the Owner and Group are 'root' which is the super user.

$ ls -l

Examining the permissions for the file sudorootfile.txt, 'rw-r--r--', we see:

rw-User 'root' has read and write but not execute.
r--Group 'root' has read but not write nor execute.
r--Everyone else (includes you) has the same as group 'root', i.e., read but not write nor execute.

Examining the permissions for the directory 'sudorootdir', 'rwxr-xr-x', we see:

(Remember, execute is required on all folders)

rwxUser 'root' has read, write & execute.
r-xGroup 'root' has read & execute but not write.
r-xEveryone else (includes you) has the same as group 'root', i.e., read & execute but not write.

Also look in Files. Both sudorootdir and sudorootfile.txt have a lock icon which means they are 'read-only' to you. However, they are 'read/write' to root.

If you right-click on the file or directory and choose and then the Permissions tab, you can see the permissions in a not-cryptic fashion.

You can remove sudorootdir and sudorootfile.txt without sudo but you will be prompted to confirm.

$ rm sudorootdir

Using sudo will work without asking.

$ sudo rm sudorootfile.txt

Now let's use sudo in a real-life situation you will encounter as a web designer or developer. In a later part of this series you will learn how to create a virtual host. Part of that process is editing the 'hosts' file which is owned by root. First we will try to edit the file without sudo.

$ gedit /etc/hosts

Add a space or any other non-destructive change. We will not saving the file so it is safe. Press Ctrl+q to start closing the file (don't save it!). Note that the file is read-only as show in the title bar and when you try to close the file. Close the file without saving it.

Now re-open the file using sudo.

$ sudo gedit /etc/hosts

Make the same edit and begin closing the file (don't save it). Observe that the file is now read/write. Close the file without saving it.

$ chown - Change Owner & $ chgrp - Change Group

After completing the steps below, please read FilePermssions: Understanding and Using File Permissions. The benefit to your ability to work on a Linux system will be enormous.

Setup sample files.

$ sudo touch rootfile.txt
$ sudo mkdir rootdir
$ ls -l

You can see that Owner and Group Owner are both 'root'.

In the below lines, use your user name in place of userName.

$ sudo chown userName rootfile.txt
$ ls -l

Owner has been change to your user name.

Now change the group.

$ sudo chgrp userName rootfile.txt
$ ls -l

Group Owner has been change to your user name.

You can change both at the same time.

$ sudo chown userName:userName rootdir
$ ls -l

$ chmod - Change Permissions

You can change permissions using letters or numbers. Numbers are quicker, but letters are easier if you are new to permissions, so I will use letters in this example.

First, let's setup a sample file, still working in ~/workspace/samplefiles. We will use a command with '*' which is called a wildcard. It will delete everything in the current directory.

$ rm -r *
$ touch permissions1.txt
$ ls -l

Give the owner execute permission. 'u+x' in the below command means 'add x to u' where x=execute and u=user.

$ chmod u+x permissions1.txt

Remove execute permission from the owner. In this case 'u-x' means 'take away x (execute) from u (user)'.

$ chmod u-x permissions1.txt
