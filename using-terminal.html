I grew-up on Windows. My computing life started less than a year before Windows 3.0 and I did little more than use word processors such as WordStar and WordPerfect. The only command line commands I knew were for listing, moving, renaming and deleting files, and I did very little of that.

Can you be a programmer and not know the command line? I nearly did. From Windows 3.0 to three years ago I developed Windows applications and I only rarely used the comand line. So yes, you can get by with very little command line savy, but perhaps not on Linux.

But that was then and I think it a bit unusual to not be good with a command line, especially in the Linux world. In fact, I'm not sure that it is possible to completely avoid the command line on Linux.

Using the command line intimidates some people. However, it just takes some getting used to, and it will open-up some possibilities. If you are one of those intimitated by the command line just stick with it. Eventually it will become intuitive

I use Files (aka File Manager) for many tasks, especially moving/renaming files that I have permission to do so with. I use the command line when installing software requires it and when working with files that are owned by the system. So on a continium of always using command line to always using GUI tools, I'm somewhere in-between but close to the GUI end.

Whereever you see $ type into terminal the text that follows.

For example, if you see $ ls -la, you type into terminal "ls -la" exactly as you see it with particular attention to capitalization and spaces.

  1. Everything you do in Terminal is case sensitive. If something doesn't work be sure to double check the case.
  2. You can open multiple tabs in terminal by pressing Ctrl-t.
  3. Spaces in file names require quotes. It is easiest to not put spaces in your file names.

Use a Cheetsheet

It will help to have a cheatsheet of commands. This can be a text file or it can go into a program such as Evernote or OneNote. Keep it simple so you can add to and refer to it quickly. A cheetsheet listing of all the commands used in this guide is at the bottom of the post.

Getting Started

Launch Terminal

  1. To launch Terminal press Super, type 'terminal' and press Enter.

Terminal will be launched with the current directory as you home directory. Let's dissect the command prompt.

karl @ karl-g3 : ~ $

karl User name
@ "at"
karl-g3 Machine/computer name
: "colon"
~ "tilda" is a short-hand for "home"
$ Indicates commands will be executed with your privildeges and not super user privilidges (which would be indicated with '#').

Setup your windows

If you have the screen real estate then open Terminal & Files side-by-side so that as you type commands into Terminal you can see the changes in Files.

The First Command "ls"

"ls Command"

$ ls

This will show the contents of you home (~) directory.

You can add options to the ls command. The most common are -l and -a

$ ls -l

"-l" (lower case L) shows a more detailed list view of the directory contents. We will analyze the parts of this detail shortly.

$ ls -la

"-a" shows hidden files.

ls -l, ls -la Output explained

For the first 10 characters, the individual characters have the following meaning

Definitions

There is a great deal of detail printed out by the ls command. We will go through it piece by piece. Some of the individual characters have specific meaning as list below.

Character Meaning
- (hyphen) When in the first position it indicates that the item is a file.
When in any of the following 9 positions it means 'the permission is not granted'.
d When in the first position it indicates that the item is a directory (aka folder)
r Read permission. The user can see and open the file.
w Write permission. The user can edit the file.
x Execute permission. The user can execute the file it is a program or other executible.
l Symbolic link.

Output Parts

- rw- rw- r-- 1 karl karl 0 May 28 08:50 test1.txt

Filetype Owner permissions Group permissions Other permissions Link count
Owner Group File size Last access date/time Filename

It is confusing to have a user named 'karl' and a group named 'karl' in the example. I'll try to clear that up below. If after that it is still causing confusion, leave a comment and I'll find a better approach.

Explanation of Output Parts

Characters 2 through 10 are groups of 3. Each group of 3 applies to a specific type of system user. Possible values for the positions with a group of 3 are:

Position 1 of 3"r" for read or "-" for 'does not have read premission'
Position 2 of 3"w" for write or "-" for 'does not have write permission'
Position 3 of 3"x" for execute of "-" for 'does not have execute permission'
RefPartMeaning
A-"-" (hyphen) indicates it is a file and not a directory. This position could also be "d" for directory. There are other designators that can exist in this position. See the references at bottom of post for more detail.
Brw-The first grouping of three (characters 2-4) is the permissions for the user. In this case user karl has read & write but not execute.
Crw-The second grouping of three (characters 5-7) is the permissions for the group. In this case, same as previous, the group karl has read & write but not execute.
Dr--The third grouping is the permissions for everyone else. In this case everyone other than user karl and members of the group karl, have read but not write and not execute.
E1The number of links to this file. An explanation of this is out of scope for this post.
Fkarl"Owner" of the item who will be an individual user account, in this case I am logged-in as karl.
Gkarl"Group owner" of the file or item which will be a group. In this case no group owns the file so karl is listed even though it is not a group.
H0This size of the item. This is an empty text file so this is showing as zero.
IMay 28 08:50The date & time.
Jtest1.txtThe name.

$ rm - Removing files and folders

There is a rmdir command but I don't find it necessary. You can remove (aka delete) any file or folder using the rm command along with a couple of options: -r for 'recursize' and -f for force.

-r (recursive) will drill down the directory structure and remove everything below the specified node

-f (force) ignores nonexistent files, missing operands and never prompts the user. What that really means is it will silently go about removing things and not present any cautions or other information to you.

Follow these steps to learn how rm works

$ cd ~/workspace/samplefiles
$ mkdir emptydir
$ mkdir nonemptydir
$ cd nonemptydir
$ > file1.txt
$ > file2.txt
$ mkdir innerdir
cd ..

Now that our sample data is setup, let's try removing some stuff

rm by itself will not work for an empty or non-empty directory. rmdir will work for both empty but not non-empty directories. I'll show you how to do both with one command.

Let's try rm to see what happens.

$ rm emptydir
$ rm nonemptydir

Now try -f wihtout -r.

$ rm -f emptydir
$ rm -f nonemptydir

That doesn't work either. However, in both cases, -r will.

$ rm -r emptydir
$ rm -r nonemptydir

rm also works on files.

$ cd ~/workspace/samplefiles
$ > rmfile1.txt
$ rm rmfile1.txt

$ sudo

sudo (superuser do) is a utility for UNIX- and Linux-based systems that provides an efficient way to give specific users permission to use specific system commands at the root (most powerful) level of the system. Sudo also logs all commands and arguments. Source: TechTarget

So far we have been working with files owned by the currently log-in user (you). If you want to work with files and directories that you don't own you need to use sudo. Let's create both a directory and a file with sudo and then view the results.

$ cd ~/workspace/samplefiles
$ sudo mkdir sudorootfiles
$ sudo > sudorootfile.txt
$ ls -l

You will be prompted for your password, the same one you log in with.

Performing a directory listing will show that both the Owner and Group are 'root' which is the super user

$ cd ls -l

I left the error messages in the above window on purpose. They appear whenever you launch gEdit form Terminal. I assume there is a fix for it but I have been using gedit for years and have just lived with the errors with no ill-effect.

Also look in Files. For both sudorootdir and sudorootfile.txt you will see a lock icon. This means that they are 'read-only' to you. They are 'read/write' to root.

You can remove sudorootdir and sudorootfile.txt without sudo but you will be prompted to confirm. Let's try on sudorootdir.

$ rm sudorootdir

However, if you try to edit sudorootfile.txt without sudo you will not be able to save it because it is read-only to you.

$ gedit sudorootfile.txt

Make a change to the file contents and then Ctrl+q.

As you can see in the accompaning picture, 1) the titlebar says [Read-Only], 2) the Save button is disabled and 3) when you quit you are offered Save As... instead of Save

Now let's give that a try in a real life scenario which you will meet later if you are following the series when creating a virtual host.

$ gedit /etc/hosts

Note that the file is read-only.

Press Ctrl+q

$ sudo gedit /etc/hosts

Now the file is read/write.
