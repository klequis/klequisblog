Using Git with GitHub Quick Start

This guide will show you how to install Git and then use Git with GitHub. This is not a complete Git tutorial but is a very good start for someone who has not used Git before. There is a very good free online book for using Git. If you are new to Git, reading at least the first three chapters will be well worth your time. See the 'See Also' section at the bottom of the post for a link to the book.

There are different ways you can create a Git repository and work with it. The Git/GitHub workflow in this tutorial is one possible process, and one that I think it is easy for a new Git/GitHub user to follow and learn from. Long-term, this is not the way to use Git. As I mention at the end of this tutorial, there is much more to Git and more to learn before you can effectively manage the software life-cycle and collaborate with other developers.

Is Git is Hard?

Once you understand the basics of Git it will no longer seem hard. This tutorial will get you on you way. We will use only **five** Git commands to accomplish a complete Git workflow. The commands are:

$ git clone
$ git status
$ git add
$ git commit
$ git push

Branching

I left branching out of this tutorial in order to keep it simple. If you were working solo you could use Git without branching, but I don't recommend it. Branching is one of Git's most powerful features and you should set about learing it as soon as you are comfortable with the concepts in this turotial. See the links at the bottom of the page for resources to learn from.

Why Use the Command Line

There are GUI/visual tools for Git and I do not in any way discourage you from using them. However, when you learn to use Git via the command line you get a fuller understanding of Git. Once you learn Git via the command line you can they adopt a visual tool, but it is very possible you will not feel the need to do so. Additionally, if you are following the **series** then you are working on a Linux machine. While you can use Linux with very little use of the command line, everywhere you go for help, for example StackExchange or Ask Ubuntu forums, the suggested solution most often envolves using the command line.

If you don't know how to use the command line you can still follow this tutorial. Just type in the commands exactly as show and it will work. However, the tutorial **this tutorial** is good preperation.

Definitions

Like many specialized subjects, Git has its own vocabulary and understanding Git well enough to make use of it requires that you learn its vocabulary. Here are some definitions to help with the next steps.

Working Directory 'Working directory' is used to refer to files in the project's folder that are currently in a state other than committed. It contains all the files from one version of the project. In our example, the working directory is ~/workspace/git-github-tutorial. It is also the project's root directory.
Staging Area The 'staging area' is not actually an 'area'. It is a file that tracks files that have been added to the staging area with $ git add.
Untracked (file) An untracked is a file in the working directory that has not been added to Git.
Tracked (file) A tracked file is a file that has been added to the Git repository with $ git add.
Staged (file) A staged file is ready to be added to Git. It is in the 'staging area' but has not been 'committed' yet. A file is staged with the $ git add command. You can use $ git add on an untracked or modified file.
Committed (file) A staged file that has moved to a committed state by using $ git commit.
Modified (file) A modified file is a tracked file that has been changed since it was last committed.
Repository ("repo") A repository can be thought of as a Git database. It is a complete copy of your project with all history.
Local Repository The 'local repository' is on your local machine in a /.git directory. In our example the repository is ~/workspace/git-github-tutorial/.git. The '.' before 'git' in .git means it is a hidden directory. To see it using Terminal, use $ ls -la where -a means 'show hidden files'.
Remote Repository The remote repsitory is a repository on a remote server. In our example, the remote repository is the one we created on GitHub. GitHub is not the only place where a remote repository can reside.
Remote A remote is a name that holds the URL to the remote repository. You can think of it as a variable that holds a string where the string is the URL to the remote repository, which in our case is GitHub
Branch Git is like a tree. You can add branches to your tree. Each branch is a complete version of your project. The default branch is called 'master'. Take a look at your GitHub home page and find the dropdown that says 'Master'. Currently, your repository only has one branch, i.e., Master

Document Life-cycle

We will be using the commands $ git add & $ git commit to move the files through their life-cycle. Don't worry about what the commands mean just yet. I'll go into them in more detail later.

Untracked Staged Committed
Modified Staged Committed
  • In the output of $ git add modified and untracked files are in red text.
  • When you add a new file to the working directory it is an 'untracked' file.
  • When you modify a previously committed file it becomes 'modified'.
  • When you include a modified or untracked file in the $ git add command it becomes 'staged'.
  • When the $ git commit command is executed, all staged files become committed. Commited files are not show in the output of $ git status.

Pushing to GitHub $ git push

To move you files to GitHub use $ git push. We will first push our project changes to GitHub and then dive into an explanation of doing so.

The syntax of $ git push is:

git push [remote-name] [branch-name]

Since our remote-name is 'origin' and our current branch-name is 'master', execute the push command as follows:

$ git push origin master

When prompted, enter you GitHub user name and password.

That's it, your chnages are already in the GitHub repository! Take a look at your project's GitHub page to confirm the files are added.

It is worth taking a moment to realize that so far we have only used 3 commands, add, commit and push.

pic.015.01.git.push.png
